Summary

The WDAqua project aims at developing research for Question Answering systems based on structured data on the Web. Part of the Web Data comprises entities that are processable only by machines, but a large subset of the data on the Web is available as literal values that human can understand. These literal values must be used to generate the answers a human expect. Yet, the meaning of a literal can only be interpreted with respect to a data type.

In the Resource Description Framework, the data model underlying the Web of Data, literals are composed of a UNICODE string (the lexical form), a datatype IRI, and optionally, when the datatype IRI is rdf:langString, a language tag.

Any IRI can take the place of a datatype IRI, but the specification only defines the precise meaning of a literal when the datatype IRI is among a predefined subset.

It may be interesting to define custom datatypes, as does the DBpedia project. Literals with custom datatypes could advantageously represent classical structures such as vectors and matrices, measurements.

"4.3 + 3i"^^<https://example.org/complex> . 
"(-4, 1)"^^<https://example.org/vector> . 
"( ( -4, 1 ) , ( 3, 2 ) )"^^<https://example.org/2dmatrix> . 
"darkturquoise"^^<https://example.org/css3-color> . 
"25 cm"^^<https://example.org/length> .

Yet, their support by RDF processors is rare and implementation specific. When an RDF processor encounters a RDF literal with a new unknown custom datatype, it needs to somehow understand what the datatype represents. This would be interesting for instance to:

  • check that the lexical form is "valid" for the custom datatype. For example, "zmefo" is not a valid lexical form for <https://example.org/complex>
  • compare the value of two literals. For example, "darkturquoise" has the same value as "#00CED1" for <https://example.org/css3-color>
  • propose ordering of literals. For example, "25 cm" has a greater value than "50 mm" for <https://example.org/length>

Expected Results

Theoretical results

  • literature review about custom datatypes;

  • an RDF vocabulary to describe families of custom datatypes.

Practical results

The results will show the interest of custom datatypes for QA. Can one infer the datatype of the expected result? Can one eliminate false results, or speed up the search?

A subset of the following:

  • defining a library of custom datatypes

  • a tool for editing custom datatypes

  • extending a classical RDF processor so that it understands new custom datatypes on-the-fly
